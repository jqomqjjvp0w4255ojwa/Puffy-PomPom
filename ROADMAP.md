## 1. Monday（即時討論劇情的對象）

**動機**：不想因為看不懂糰糰的行為（比如為什麼不吃飯），就要把整段對話手動搬去給別的 AI 看。想要一個能直接讀取「當前頁面」的旁觀者，跟玩家一樣是個看客，不必懂糰糰的全部歷史。

**形式**：
- UI 比照租車網站的「網站小幫手」模式：在便利貼（筆記）icon 旁邊加一個笑臉／對話泡泡 icon，放在書頁外面左下角，避免擋到既有版面。
- 人設可自訂，目前構想：叫 **Monday**，性格厭世。

**技術構想**：
- 用便宜的 Gemini（Flash）。
- 不需要玩家手動複製對話，而是後端提供一個「當前狀態快照」端點（糰糰狀態、最近摘要、房間環境等），讓 Monday 自動讀取作為 context。
- 規模要控制：用既有的 day-summary / week-summary 機制做精簡上下文，避免把所有原始日誌塞進 prompt。

**待決定**：
- Monday 要不要記住「上次聊到哪」，還是每次重新讀取當前狀態（無記憶、純旁觀者）？
- 觸發機制：常駐泡泡，還是點開才載入資料？

## 2. 照片分享（變相日記 + 圖像認知小遊戲）

**動機**：想跟糰糰分享照片，本質上是把這個系統變成自己的視覺日記。

**現實限制**：技術上可行（多模態模型能「看」照片），但成本上不行（vision API + 圖床費用）。

**保留的折衷構想（尚不實作）**：
- 做成**獨立頁面的小遊戲**，而不是直接整合進日常生態系統。
- 玩家上傳照片 → 由 Gemini 判斷圖片內容 → 把「認知結果」餵給糰糰 → 糰糰吐出內心飄渺跳脫的想法，或展開一段「糰糰（腦內）走進這張照片的小冒險」敘事。
- 重點是趣味性與不確定性，每次結果可能都不一樣，類似抽卡/ 隨機小品文。

**待決定**：
- 圖床方案（GitHub repo 直接存 / 外部圖床服務）。
- 是否要把這些「冒險敘事」存檔成一種特殊日記類型，回顧時能重新看到。

## 3. 資源使用的軟性後果機制（原構想：經濟系統）

**真實動機（重新定調後）**：起點其實是玩家自己現實生活中的焦慮感——比如「冰箱門開太久，現實裡電費會爆，所以不敢一直開」。原本想做「經濟系統」，但討論後發現核心需求不是金錢/購物模擬，而是**把現實裡資源消耗的潛意識焦慮，投射成糰糰房間裡的一種「軟性後果」**。

**設計原則（重要）**：
- **不做數字儀表板**（不需要「電費：128元」這種介面），而是用**敘事後果**取代量化系統——玩家不需要看到具體數字，只需要「感覺到」某些行為有重量。
- 例如：冰箱開太久 → 糰糰可能會抱怨冷、某個東西可能會壞掉、房間氣氛可能變了，而不是跳出一個帳單。
- 這個方向跟既有的「猜測優先」世界觀是相容的：玩家依然看不到後台的數值運算，只看到行為與敘事上的回饋。

**糰糰本身已有的樸素交換邏輯（線索）**：
- 玩家給「很強的風」，糰糰就幫忙「趕走三顆灰塵」——顯示糰糰心智中本就存在某種「付出/回饋」邏輯，是物理現象之間的隱喻交換，不是金錢。

**好玩的衍生想法（先記錄，未評估）**：
- 能不能想辦法「從小黑影身上弄點錢/資源出來」——小黑影本身設定上比較陰暗、感覺可以做點「犯罪」的事。如果要往這方向發展，玩家跟小黑影的互動可以是這個「軟性後果系統」裡比較刺激／灰色地帶的支線（比如某種有風險、有代價的資源來源，跟跟糰糰的正當交換邏輯形成對比）。

**需要先討論清楚才能往下走的問題**：
1. 哪些「現實焦慮」值得映射進房間？（電費/開冰箱只是一個例子，還有沒有其他類似的「我在現實會緊張的小事」？）
2. 後果要怎麼呈現？是糰糰的台詞/心情變化，還是環境本身的變化（比如東西變舊、變壞）？
3. 玩家要不要有「主動觸發」的入口，還是純粹被動承受（開了就開了，看糰糰怎麼反應）？
4. 小黑影的灰色資源支線要不要做成正式機制，還是先留著當有趣的腦洞？

**目前判斷**：方向已經比之前清楚很多，不再是「最大最模糊」的一塊，但仍建議在第 4 點（糰寶庫）有具體雛形後再回頭，因為「後果」很可能要透過糰糰的窩/收藏狀態變化來呈現，會用到同一套底層機制。

## 4. 糰寶庫（收藏癖系統）＋ 糰窩 —— 糰糰自己的「文明」

**動機**：白糰糰會替物品賦予感情，喜歡到處撿東西收藏。這不只是個寵物收集清單，而更像是糰糰自己在經營一個玩家看不懂、看不到全貌的隱性世界——「糰糰在我房間不只是存在，而是在建立一個我不明白的文明」。

**目前構想的世界觀（重要！這是核心定調）**：
- 糰糰有一個**看不見的背包（毛層）**，可以從裡面掏出東西進行交換或使用。
- 糰糰有**不同的窩**，每個窩裡也有自己的寶藏。
- 糰糰能**自己調度資源**——不是玩家在管理庫存，而是糰糰在背後自主決策、搬移、交換、收藏。
- 類比上更接近「文明 / 世紀帝國」式的資源調度系統，而不是單純的寵物收藏清單。

**喜歡撿的東西清單**（具體素材，已有）：
- 冰棒棍
- 絲襪纖維
- 紙條
- 塑膠環
- 小襪子
- 奇怪石頭

**糰窩機制**：
- 記錄「進行過築窩儀式」的地點。
- 糰糰太久沒踏足某個窩，該窩會**消失**（時間衰減機制）。

**AI 互動迴圈（這是讓系統「活起來」的關鍵，而不只是資料庫）**：
- AI 生成糰糰行為時，會參考他的物品庫、偏好、目前窩的狀態，影響決策（例如：他有沒有喜歡的毛？這週纖維夠不夠？要不要把某個東西換出去？）。
- 當糰糰撿到一個**從沒被定義過的新東西**（例如「阿里巴巴的背毛」這種莫名其妙、聽都沒聽過的物品名），AI 要能**自己判斷、自己賦予臨時屬性**，並存進寶庫——而不是出錯或忽略。這意味著物品系統要設計成「開放詞彙」，不能是寫死的固定清單。
- 玩家通常看不到糰糰調度資源的全貌，只能透過糰糰展示出來的行為／敘述片段去推測——維持「文明感」與神秘感。

**技術上可參考**：
- 跟既有的 `hidden-stats.json` / `events.json` 的時間戳衰減模式很像，技術上可以複用類似的機制（如「太久沒踏足會消失」）。
- 物品需要「開放生成」的能力，不能只靠寫死清單比對，AI 要能即興賦予新物品意義與屬性。

**待決定**：
- 物品/窩的資料結構（這部分等真正動工時再設計）。
- 玩家是否有任何「主動介入」的入口，還是純粹只能旁觀、事後從敘事片段拼湊理解？

## 5. 成就系統（手繪/像素照片收藏，低調觸發）

**動機**：不是劇情向的視覺小說，而是想要「達成某些條件可以收集糰糰照片」＋「隨機小彩蛋事件」，例如：糰糰爬上床、插上竹籤、宣布這裡是他的窩，玩家不能上來了。

**核心設計原則**：
- **不彈窗、不強調「成就達成」**——低調寫入筆記或紀錄裡一筆，玩家自己翻到才會發現，符合糰糰世界「猜測優先」的調性。
- 觸發來源兩種：
  1. **數值型**：靠既有 hidden-stats（熟悉度／好感度等）算出的條件達成（例如熟悉度首次破某個門檻）。
  2. **隨機彩蛋型**：獨立一份條目清單，給低機率 roll，類似 events.json 現有的特殊事件結構，但不需要數值門檻、純隨機觸發。
- **照片**：玩家未來會自己手繪或做成像素動圖，不靠 AI 生圖。系統現在只需要存「成就 id + 時間 + 觸發當下的文字描述」，圖片之後再手動對應補上即可，不卡現在的開發。

**儲存位置（已決定）**：複用既有「筆記」分頁的書櫃圖鑑系統（`data/fragments.json` 的 chapter/items 結構），開一本新書冊（例如《糰糰相簿》），成就照片＝這本書裡的條目。差異只是觸發方式從關鍵字隨機機率，改成數值門檻/彩蛋直接解鎖；條目多留一個圖片欄位給之後手繪/像素圖用。不需要另開新頁面或新 UI。

**已決定**：成就/彩蛋的文字記錄不進主敘事資料流（不影響 day/week summary 與 AI 記憶），只存成獨立的書冊條目。若彩蛋本身是「糰糰做的一件事」，可以留一筆極簡場景描述讓玩家當下看到，但跟「成就解鎖」這個後設標記分開存。

**待討論的細節**：
- 數值型成就的門檻清單（哪些 hidden-stats 條件值得設成就點）。
- 隨機彩蛋的條目清單與大致機率。
- 資料結構：`world.achievements`（陣列，記 id/time/text）+ 對應 unlock 邏輯放在 tick 裡還是事件系統裡。

## 6. 寵物監視器（數值驅動的視覺化糰糰）

**動機**：目前整個系統是純文字敘事，這個構想要做一個固定視窗、可左右平移的「監視器畫面」，把糰糰目前的各種數值即時轉換成視覺呈現，而不只是文字描述。

**構想（目前偏簡單版本）**：
- 畫面裡的糰糰由數值計算呈現幾種狀態，數值包含物理性與抽象兩類：
  - 物理性：例如房間 X 軸座標決定糰糰在畫面中的左右位置。
  - 抽象：例如心情高→毛變澎（變大顆）、飢餓→來回走動。
- 玩家可以左右平移視窗去找糰糰目前在房間的哪個位置。
- 參考來源：使用者另一個倉庫「iceland」的 `index.html` 有類似的圖像呈現方式可以參考（**目前這個 session 的 GitHub 存取範圍只開放 puffy-pompom，看不到 iceland 倉庫內容，需要使用者之後直接貼程式碼/截圖或開放該倉庫存取才能具體比對做法**）。

**長期野心（先記錄，不評估）**：
- 想像中最終要做成更複雜的「生態箱」——不只是顯示幾個簡單狀態，可能要呈現環境互動、多種行為動畫等更豐富的畫面。

**跟糰糰相簿系統的連動構想**：
- 監視器是「持續呈現」，相簿（第5點的成就書冊）是「精選快照」。
- 監視器畫面裡出現的某些特殊瞬間（例如誇張姿態、罕見行為組合）可以做成相簿解鎖的判定條件之一，把兩個系統串起來。

**技術門檻評估**：
- 比現有任何功能都重，需要額外的 2D 渲染層（CSS 或 Canvas），把數值映射成視覺參數（位置/體積/動作頻率），是目前 ROADMAP 中技術量級最大的一項。
- 建議先做最簡化版本驗證「看起來像糰糰」這個核心樂趣點（例如純 CSS 變形的圓形毛球＋幾個數值映射），不要一開始就衝著「生態箱」的複雜度去設計。

**待決定**：
- 簡化版的視覺呈現方式（CSS 變形 vs. 簡單 sprite 動畫 vs. 參考 iceland 倉庫的既有做法）。
- 數值→視覺參數的映射清單（先列 3-5 個最有趣的數值對應關係即可，不用一次做全）。
- 跟相簿系統串接的判定邏輯細節（等兩個系統都有雛形再談）。

## 排序建議（暫定，可隨時調整）

目前認為**第 4 點（糰寶庫／文明系統）**概念最成熟、最契合既有世界觀，可以優先深入設計。
第 3 點（經濟系統）很大程度依賴第 4 點的物品/資源庫存基礎，建議在第 4 點有具體雛形後再回頭討論。
第 1 點（Monday）技術門檻最低，可以隨時找空檔先做。
第 2 點（照片小遊戲）目前已決定先擱置、僅記錄構想。
第 5 點（成就系統）門檻不高，可以跟第 1 點一起排在前面先做。
第 6 點（寵物監視器）目前技術量級最大，建議等第 4、5 點有雛形、且能參考到 iceland 倉庫做法後再評估時程；可以先做最簡化版本驗證樂趣點，不用一次衝著「生態箱」的複雜度設計。
